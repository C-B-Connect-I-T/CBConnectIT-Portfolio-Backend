# Service and Project media in MediaFiles

Service and Project media are modeled as `MediaFiles` records (not URL columns), with asymmetric rules enforced by schema and application validation: **Project must always have both `IMAGE` and `BANNER`**, while **Service must always have `IMAGE` and may omit/remove `BANNER`**. Project create requires both files and Project update may replace either file but does not support banner-removal flows; Service create requires `IMAGE` and allows `BANNER` to be omitted. Media parsing is handled by a generic multipart helper instead of the testimonial-specific helper. No backfill is performed because release includes a database reset, so historical Service/Project media in existing environments is intentionally discarded.
